What it tracks

Six metrics that matter

Everything is labeled consistently so you can slice and compare across providers, models, and product features.

📊
Request Volume
Total requests processed, broken down by provider, model, feature, and status.
⏱
Latency
Full histogram: p50, p95, and p99 per model. See which providers are slow.
📥
Input Tokens
Total prompt tokens consumed. Input cost is usually the larger driver.
📤
Output Tokens
Total completion tokens generated. Helps understand actual model output behaviour.
💵
Estimated Cost
Approximated USD spend per request based on hardcoded per-1K token pricing. Not billing-grade.
🔴
Failures
Error counter + error rate panel. Simulator injects ~8% failure rate so the panel has data.
How it works

Five steps, no magic

The full data path from HTTP request to Grafana panel.

01
Request hits FastAPI
A client sends POST /chat with a provider, model, feature, and prompt payload.
02
Simulator generates realistic output
FakeLLMSimulator.generate() produces realistic latency variance, token counts, and occasional errors — without touching a real LLM API. No keys required.
03
Metrics recorded with labels
The app updates Prometheus counters and histograms, tagging each with provider, model, endpoint, feature, and status.
04
Prometheus scrapes /metrics every 5 seconds
Prometheus polls the /metrics endpoint and stores each data point as a labeled time series.
05
Grafana renders the dashboard
Grafana queries Prometheus using PromQL and renders the prebuilt LLM Cost + Latency Monitor dashboard. Auto-provisioned — open it immediately after startup.
Architecture

How the pieces connect

Three services wired together by Docker Compose. No manual configuration needed.

The POST /chat endpoint runs the simulator and records metrics. GET /metrics exposes them to Prometheus. Grafana reads Prometheus. All three start with one command.

Beginner primer

What Prometheus and Grafana actually do

If you haven't used these before, here's the plain-English version.

Prometheus
Metrics collection & storage

Your app exposes a /metrics endpoint that returns numbers in a specific text format. Prometheus polls that endpoint on a schedule — every 5 seconds here — and stores each value as a time series: metric name, labels, value, timestamp.

It also gives you PromQL, a query language for computing rates, percentiles, and aggregations over stored data. That's how you ask "what was the p95 latency for model X over the last 5 minutes?"

Grafana
Visualization

Grafana connects to Prometheus as a data source. You write PromQL queries, and Grafana turns the results into charts and dashboards. It doesn't store any data itself — it reads from Prometheus and renders it.

In this project the dashboard is pre-built and auto-provisioned, so you open Grafana and it's already configured. No setup required after docker compose up.

In this project: FastAPI exposes /metrics → Prometheus scrapes it every 5s and stores the time series → Grafana queries Prometheus with PromQL → you see a live dashboard. Docker Compose wires all three together automatically.

Quick start

Up and running in ~2 minutes

You need Docker with Compose. That's it.

1
Clone the repo
git clone https://github.com/srimonchaari/llm-observability.git
cd llm-observability
2
Start all services
docker compose up --build
Starts FastAPI on :8000, Prometheus on :9090, and Grafana on :3000.
3
Send a request
curl -X POST http://localhost:8000/chat \
  -H "Content-Type: application/json" \
  -d '{
    "provider": "openai",
    "model": "gpt-4o-mini",
    "feature": "support-bot",
    "endpoint": "/chat",
    "prompt": "Summarize customer issues",
    "max_output_tokens": 120
  }'
4
Open Grafana
Go to http://localhost:3000, log in with admin / admin, and open the LLM Cost + Latency Monitor dashboard. It's already provisioned.

Sample PromQL queries

Error rate (5-minute window) 
sum(rate(llm_errors_total[5m]))
  /
sum(rate(llm_requests_total[5m]))
p95 latency by model 
histogram_quantile(0.95,
  sum(rate(llm_request_latency_seconds_bucket[5m])) by (le, model)
)
Estimated cost rate by model 
sum by (model) (rate(llm_estimated_cost_usd_total[5m]))
Limitations

What this isn't

This is a v1 demo. Here's what you should know before plugging it into anything serious.

⚠
Simulated traffic only. No real LLM API calls are made. The simulator produces realistic-looking numbers but they don't correspond to actual provider behaviour.
⚠
Cost is estimated, not exact. Pricing is hardcoded per-1K token rates. Real provider pricing changes, varies by tier, and includes factors this doesn't model.
⚠
Not billing-grade. Token counts are approximated from word count, not a real tokenizer. Do not use these numbers for invoicing or cost attribution.
⚠
No authentication. Grafana runs with default credentials. Prometheus has no access control. This is a local dev stack, not a production deployment.
⚠
v1 scope. No structured logging, no distributed tracing, no alerting rules, no per-tenant labels. These are planned for v2.
