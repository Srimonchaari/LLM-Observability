What is this project?

LLM Observability Dashboard is a demo observability stack for LLM-backed APIs. It wraps a FastAPI app with Prometheus metrics and ships a prebuilt Grafana dashboard that visualizes those metrics in real time.

The app uses a local simulator — no real LLM API keys are needed. You get a working metrics pipeline out of the box: send requests, watch counters and histograms update, and explore the Grafana dashboard.

Why it was built

LLM APIs are opaque by default. You send a prompt and get a response, but there's no built-in visibility into cost, latency, or failure rate — especially across multiple providers or models.

This project shows how to instrument an LLM API the same way you'd instrument any production service. The patterns here — metrics, labels, dashboards, PromQL queries — apply directly to real deployments.

How it works

  1. A request hits the FastAPI POST /chat endpoint.
  2. The app calls FakeLLMSimulator.generate(), which produces realistic latency, token counts, and occasional failures — without making real API calls.
  3. The app records Prometheus metrics using consistent labels (provider, model, endpoint, feature, status).
  4. Prometheus scrapes /metrics every 5 seconds and stores the time series.
  5. Grafana reads Prometheus and renders the prebuilt dashboard.

The simulator injects an 8% failure rate so error-rate panels in the dashboard have something to show.

Key metrics tracked

Metric Type Description
llm_requests_total Counter Total requests processed
llm_errors_total Counter Total failed requests
llm_input_tokens_total Counter Total input tokens consumed
llm_output_tokens_total Counter Total output tokens generated
llm_estimated_cost_usd_total Counter Cumulative estimated cost (USD)
llm_request_latency_seconds Histogram Request latency (buckets: 50ms–21s)

All metrics include labels: provider, model, endpoint, feature, status.

Sample PromQL queries

Error rate:

sum(rate(llm_errors_total[5m])) / sum(rate(llm_requests_total[5m]))

p95 latency by model:

histogram_quantile(0.95,
  sum(rate(llm_request_latency_seconds_bucket[5m])) by (le, model)
)

Estimated cost rate by model:

sum by (model) (rate(llm_estimated_cost_usd_total[5m]))

Prometheus & Grafana — plain English

If you haven't used these tools before:

Prometheus
Metrics collection & storage

Your app exposes a /metrics endpoint with numbers. Prometheus scrapes that endpoint on a schedule and stores each value as a time series — metric name, labels, value, timestamp. It also has a query language (PromQL) for aggregations like rates and percentiles.

Grafana
Visualization

Grafana connects to Prometheus as a data source. You write PromQL queries, and Grafana turns the results into charts, graphs, and dashboards. It doesn't store any data itself — it just reads from Prometheus and renders it.

In this project: FastAPI exposes /metrics → Prometheus scrapes it every 5s → Grafana reads Prometheus → you see a live dashboard. Docker Compose wires all three services together automatically.

Setup

Requirements: Docker with Compose.

  1. Clone the repo 
    git clone https://github.com/srimonchaari/llm-observability.git
cd llm-observability
  2. Start all services 
    docker compose up --build
  3. Send a request 
    curl -X POST http://localhost:8000/chat \
  -H "Content-Type: application/json" \
  -d '{
    "provider": "openai",
    "model": "gpt-4o-mini",
    "feature": "support-bot",
    "endpoint": "/chat",
    "prompt": "Summarize customer issues",
    "max_output_tokens": 120
  }'
  4. Open Grafana

    Go to http://localhost:3000, log in with admin / admin, and open the LLM Cost + Latency Monitor dashboard.
